# Matt Editor vs MSBT Legit Builder Logic Audit

## 1. Executive Summary

The current MSBT Python Legit Builder and the hosted Matt editor solve related but different problems.

The Python Legit Builder is a guided, root-and-slot based builder. It is best kept as the fast strict builder for known-good "legit" item construction, plus a limited unlocked mode for power users who want extra options without leaving the MSBT workflow.

The Matt editor is the stronger model for advanced modded item construction. It is built around parsing, preserving, editing, and reserializing existing item serials. It understands token order, typed root/sub tokens, array tokens, missing/raw tokens, broad part maps, and relaxed construction workflows better than the current Python builder.

The main lesson is not to merge the two systems blindly. MSBT should keep the Python builder strict and predictable, while exposing the Matt editor as the advanced modded editor. Later, MSBT can adapt selected ideas from the Matt editor, especially import/rebuild, token-order preservation, better part browsing, and clearer output confirmation.

Recommended near-term direction:

1. Confirm the hosted Matt editor can generate and deliver one confirmed `@U` serial from the packaged app.
2. Label the hosted editor clearly as the advanced modded item editor.
3. Compare the same simple item generated by both builders before porting any logic.
4. Keep normal Legit Builder strict.
5. Move shared logic only after the behavior is understood and tested.

## 2. Current Python Legit Builder Model

Files inspected:

- `external_app/v22_parts_codes_fixed/external_legit_builder.py`
- `external_app/v22_parts_codes_fixed/matts_external_app_v22.py`
- `external_app/v22_parts_codes_fixed/external_serial_tools.py`
- `external_app/v22_parts_codes_fixed/resources/legit_rules_flat.json`
- `external_app/v22_parts_codes_fixed/resources/observed_working_part_options.json`

The Python builder is centered on `external_legit_builder.py`.

Core model:

- `roots()` and `all_roots()` expose buildable roots from `legit_rules_flat.json`.
- `_BUILDABLE_ROOTS` is a hard whitelist of buildable serial roots, including manufacturer and item type metadata.
- `slots(root_key)` exposes the dependency/slot tables for the selected root.
- `slot_counts(root_key, selected)` calculates min/max/current counts for each slot.
- `search_parts(root_key, text, table, limit)` returns parts for a selected root and slot.
- `is_part_allowed(root_key, selected, candidate, table)` tests whether a candidate can be added under current legit rules.
- `validate(root_key, selected, strict_comp=True)` validates the selected part list.
- `build_human(root_key, selected, level, seed, seed2)` emits human-readable serial text.
- `build_base85(root_key, selected, level, seed, seed2)` converts human serial text into `@U` using `external_serial_tools.human_to_serial`.

UI flow in `matts_external_app_v22.py`:

- Pick Type.
- Pick Manufacturer.
- Pick Root Variant.
- Optionally filter available parts.
- Pick parts inside slot cards.
- Validate.
- Build Base85.
- Give active build through the bridge.

The selected part state is normalized into compact `table:key` lines. The UI renders those lines in the "Selected Composition / Active Build" box, then validation/build read the same normalized list.

The builder output is intentionally canonical:

- `_sort_parts_for_serial()` sorts parts by slot order, dependency index, serial, and key.
- `_serial_part_token()` emits `{SubSerial}` for parts native to the current root and `{RootSerial:SubSerial}` for inherited/cross-root parts.
- `build_human()` emits one generated human serial line from the selected root and sorted parts.

This model is good for guided correctness. It is not a full token editor.

## 3. Python Unlocked/Modded Mode Model

Unlocked mode in the Python builder is not a complete advanced modded editor. It is a relaxed version of the same slot-based builder.

Current behavior:

- The same Type, Manufacturer, Root, and slot UI is used.
- The same root-centric serial generation path is used.
- Normal slot restrictions are relaxed.
- Duplicate/add-quantity behavior is allowed more often.
- `observed_working_part_options.json` adds extra known-working options to the candidate lists.
- `observed_working_slots()` can expose extra slot names.
- `observed_working_parts()` can expose known working parts that are not in the normal root slot list.
- `validate_build(..., unlock_modded=True)` treats a build as acceptable even when normal validation fails, while still reporting that rules were bypassed.
- `build_base85_external(..., unlock_modded=True)` still generates through the Python builder's human serial path and then serializes.

Important limitation:

Unlocked mode still starts from a selected root and still emits a canonical generated serial. It does not preserve arbitrary imported token order. It does not act like a raw serial/rebuild editor.

This is why it should be described as "Unlocked/Modded options for the guided builder", not as the full advanced modded editor.

## 4. Current Matt Editor Model

Files inspected:

- `external_app/v22_parts_codes_fixed/matt_editor_host.py`
- `external_app/v22_parts_codes_fixed/matt_editor_adapter.js`
- `external_app/v22_parts_codes_fixed/matt_editor/index.html`
- `external_app/v22_parts_codes_fixed/matt_editor/js/legit-builder/legit-builder.js`
- `external_app/v22_parts_codes_fixed/matt_editor/js/item-editor/domain/parse.js`
- `external_app/v22_parts_codes_fixed/matt_editor/js/item-editor/domain/parts.js`
- `external_app/v22_parts_codes_fixed/matt_editor/js/item-editor/domain/process-data.js`
- `MATT_LEGIT_BUILDER_PORT_PLAN.md`

The Matt editor is hosted locally by `matt_editor_host.py`. It serves the vendored web editor and provides local bridge/API endpoints so the editor can convert serials and deliver a confirmed final serial through the MSBT bridge.

Important host behavior:

- The editor UI is served from `external_app/v22_parts_codes_fixed/matt_editor`.
- The host intercepts serial conversion requests and routes them to local `external_serial_tools` conversion.
- `/msbt/deliver` sends only one confirmed `@U` serial to the existing bridge actions:
  - `give_serial_selected`
  - `give_serial_all`
  - `give_serial_nonhost`

The Matt editor has two relevant item-making surfaces:

- The Modded Item Builder in the item editor tab.
- The Legit Item Builder in the legit builder tab.

The Modded Item Builder has a richer import/edit/output workflow:

- Paste existing Base85 or human-readable serial.
- Parse existing item code.
- Edit item properties.
- Browse/build item parts.
- Show item parts breakdown.
- Generate human-readable item code.
- Generate serialized Base85.
- Copy or deliver the final serial through the adapter.

The Matt editor's parser preserves token order. In `parse.js`, item parts are collected from the serial body as array tokens, typed/simple tokens, and string tokens, then sorted by their original index before being rebuilt into `currentParts`.

It supports richer token forms than the Python builder UI exposes:

- Simple numeric tokens like `{87}`.
- Typed tokens like `{255:545}`.
- Array tokens like `{234:[31 3]}`.
- String tokens.

The Matt editor also contains dedicated logic for modded rows:

- `_mi_cross_manufacturer`
- `_mi_does_not_exist`
- raw missing token handling
- token order preservation for modded human serial rebuilds
- collapsing consecutive `{root:sub}` tokens into array token runs where appropriate
- deduping cross rows that serialize the same as native rows

The delivery adapter is intentionally conservative:

- It looks for the final Base85 output.
- It requires exactly one confirmed `@U` serial.
- It shows the "Serial to Send" preview.
- It refuses stale or ambiguous output.
- It posts delivery to the local host, which routes to the existing MSBT bridge.

## 5. Side-by-Side Comparison Table

| Area | Python Legit Builder | Python Unlocked Mode | Matt Editor |
| --- | --- | --- | --- |
| Primary purpose | Guided strict item build | Guided build with extra options | Advanced import/edit/rebuild/modded editor |
| Main source of truth | `legit_rules_flat.json` | `legit_rules_flat.json` plus observed options | Vendored web editor data maps and parser |
| Root selection | Whitelisted buildable roots | Same root flow | Broader item/type model |
| Part browsing | Slot-based allowed candidates | Slot-based plus observed candidates | Rich part browser and existing serial parser |
| Validation | Conservative legit rules | Reports bypassed rules | Can validate, but modded workflow is more tolerant |
| Output order | Canonical sorted order | Canonical sorted order | Preserves imported token order where needed |
| Duplicate behavior | Restricted by rules | More permissive | Designed for relaxed/modded editing |
| Unknown/missing tokens | Generally rejected or hidden | Some observed/raw support | Can preserve missing/raw tokens |
| Base85 generation | Local `human_to_serial` | Local `human_to_serial` | Host-local conversion through MSBT endpoint |
| Delivery | Generic MSBT bridge actions | Generic MSBT bridge actions | Adapter preview, then generic MSBT bridge actions |
| Best use | Normal users and strict legit builds | Intermediate users needing known-working extras | Expert item makers and imported modded serials |

## 6. Modded Item Construction Differences

The biggest construction difference is that the Python builder creates a serial from structured slot choices, while the Matt editor can parse and rebuild a serial from the token stream.

Python builder:

- Starts with a selected root.
- Looks up slots for that root.
- Adds selected parts as `table:key` entries.
- Validates counts and constraints.
- Sorts parts into canonical serial order.
- Emits a new human serial.

Matt editor:

- Can start with an existing serial.
- Parses the serial body into ordered token objects.
- Preserves token order from the original serial.
- Supports simple, typed, array, and string tokens.
- Can carry tokens even when the loaded dataset does not know what they are.
- Has explicit modded concepts for cross-manufacturer and does-not-exist rows.

This means the Matt editor is better for:

- Rebuilding known-working modded items.
- Preserving exact odd token order.
- Keeping tokens that are not fully understood by local rule data.
- Exploring combinations without forcing them through strict slot validation.

The Python builder is better for:

- Producing clean strict outputs.
- Keeping users inside known valid slots.
- Quick testing and repeatable builds.
- Avoiding accidental malformed output for normal users.

## 7. Serial/Token Ordering Differences

Python builder order:

- The Python builder sorts selected parts with `_sort_parts_for_serial()`.
- The sort key is based on root slot order, dependency index, serial value, and part key.
- This is ideal for repeatable generated output.
- It can lose the exact user/imported token order that may matter for testing modded patterns.

Matt editor order:

- The parser records the index of each token in the original serial.
- It rebuilds `currentParts` sorted by original token position.
- The modded builder has explicit helpers for preserving deserialized token order.
- It can normalize consecutive root/sub runs into array tokens when appropriate.

MSBT should not force the Matt editor's token-order behavior into the strict Python builder. Instead, the advanced editor should own token-order sensitive workflows.

## 8. Validation Differences

Python builder validation:

- Checks that a selected root exists and is buildable.
- Checks required composition/root parts.
- Checks slot min/max counts.
- Checks dependency exclusions.
- Checks component tags and part rules.
- Checks duplicate/conflict cases.
- Blocks normal-mode builds when strict rules fail.
- In unlocked mode, allows output while reporting that rules were bypassed.

Matt editor validation:

- Has a strict legit builder surface.
- Also has a relaxed modded item builder surface.
- Can parse and preserve structures that the strict Python builder should not bless as legit.
- Can keep raw/missing tokens visible instead of dropping them.
- Uses richer browser/data model behavior for editing and display.

Important conclusion:

The Python builder should keep a strict "legit" meaning. The Matt editor can own "this might work in-game" modded exploration. Mixing those meanings will confuse users and make bug reports harder.

## 9. Output/Delivery Differences

Python builder output:

- The app stores generated human output and Base85 output in Python/Tkinter state.
- Delivery uses the existing bridge actions.
- The app blocks stale generated output by checking build signatures before delivery.

Matt editor output:

- The web editor writes final human and Base85 outputs into page elements.
- The adapter reads the final Base85 output from known elements:
  - `mi_finalOutputBase85`
  - `finalOutputBase85`
  - `serializedOutput`
- The adapter requires a confirmed single `@U` serial.
- The host sends that serial to the same generic bridge delivery actions.

The adapter is the right boundary. It lets the editor keep its own build model while MSBT keeps the live game delivery path centralized.

## 10. What Python Builder Should Keep

The Python Legit Builder should keep:

- Strict normal mode.
- Type -> Manufacturer -> Root -> Slots workflow.
- Slot cards and min/max slot feedback.
- Clear validation messages.
- Canonical generated output.
- Local Base85 generation.
- Stale-output protection before delivery.
- Generic bridge delivery through `give_serial_selected`, `give_serial_all`, and `give_serial_nonhost`.
- Max passive shortcuts where they are reliable.
- Observed working options as an optional unlocked-mode expansion only.

It should not try to become the full modded editor.

## 11. What Matt Editor Should Own

The Matt editor should own:

- Advanced Modded Item Builder workflow.
- Existing serial import and rebuild.
- Token-order preserving edits.
- Array token handling.
- Unknown/raw token preservation.
- Cross-manufacturer and cross-root modded editing.
- Broad part search and data-driven part maps.
- Rich breakdown/preview UI.
- Final "Serial to Send" preview before delivery.

MSBT should host and launch it, but not reimplement every advanced editor behavior in Tkinter unless there is a very specific reason.

## 12. What Logic Can Be Adapted Later

Useful Matt editor ideas to adapt later:

- Import/rebuild workflow for Python-side bookmarks or BL4 Codes.
- A canonical "active serial to deliver" preview shared by all builders.
- Token-order aware internal state for an advanced Python-only editor, if one is ever needed.
- Better part search labels and display names.
- Better handling of raw/missing tokens in breakdowns.
- Part browser grouping by item type, manufacturer, rarity, and known source.
- Safer "compare generated serials" tooling between Python builder and Matt editor.

Do not adapt yet:

- Broad relaxed/modded acceptance into normal Legit Builder.
- Matt editor's full web UI logic into Tkinter.
- Direct delivery from editor internals without the adapter preview.
- Large browser-side data systems into the Python app unless performance and ownership are clear.

## 13. Risks

Risk: Users may confuse "modded can be generated" with "safe or legit".

Mitigation: Label the hosted Matt editor as Advanced Modded Editor and keep the strict Python Legit Builder separate.

Risk: Copying Matt editor logic piecemeal could break behavior that depends on the surrounding web app state.

Mitigation: Keep the hosted editor intact while auditing specific functions before reuse.

Risk: The Python builder's canonical sorting may produce different item strings than a known-working imported modded serial.

Mitigation: Do not use the Python builder as the only rebuild path for imported modded items.

Risk: The hosted editor may depend on large resource files and browser behavior.

Mitigation: Treat it as a packaged local web editor and validate source, EXE, and package paths separately.

Risk: Delivery bugs can become harder to diagnose if several builders deliver differently.

Mitigation: All builders should deliver through the same confirmed serial boundary and bridge actions.

Risk: The Matt editor contains legacy online/API paths.

Mitigation: Keep host-local conversion active and test offline packaging.

## 14. Recommended Implementation Stages

### ITEM-1: Confirm Matt Editor Delivery Preview Works

Goal: Verify the existing host and adapter from the packaged app.

Checks:

- Hosted editor opens from the external app.
- Modded Item Builder generates exactly one final `@U` serial.
- The "Serial to Send" preview shows the exact serial.
- Delivery refuses empty, stale, or multiple serials.
- Delivery sends through `give_serial_selected`, `give_serial_all`, or `give_serial_nonhost`.

No builder logic should change in this stage.

### ITEM-2: Add Explicit Advanced Modded Editor Entry Point

Goal: Make user intent clear.

Changes:

- Label the hosted Matt editor as "Advanced Modded Editor".
- Add short text explaining that normal Legit Builder is strict, while Advanced Modded Editor is for imported/modded/experimental serials.
- Keep the warning practical, not scary or noisy.

### ITEM-3: Compare Same Simple Item From Both Builders

Goal: Understand exact differences before porting logic.

Test:

- Build a simple pistol or class mod in Python Legit Builder.
- Build the same item in Matt editor.
- Compare human serial.
- Compare Base85.
- Compare breakdown.
- Document differences in root, flags, seed, token order, and selected parts.

### ITEM-4: Add Import/Rebuild Workflow If Missing

Goal: Let item makers start from a known-working serial.

Likely best place:

- Matt editor first.
- Python builder only if a strict rebuild path is actually needed.

Expected behavior:

- Paste existing `@U` or human serial.
- Decode locally.
- Show generated human serial and final Base85.
- Preserve token order where possible.

### ITEM-5: Move Selected Matt Editor Logic Into Shared Docs/Resources Only After Testing

Goal: Reuse knowledge without destabilizing the app.

Candidate logic:

- Part label/display metadata.
- Known token grammar notes.
- Import/rebuild comparison tests.
- Observed working part resources.

Avoid moving large browser systems until a specific Python feature needs them.

### ITEM-6: Keep Normal Legit Mode Strict

Goal: Protect the existing reliable workflow.

Rules:

- Normal mode should not accept arbitrary cross-root or raw tokens.
- Unlocked mode can remain the guided modded expansion.
- Advanced Modded Editor should own the truly relaxed path.

### ITEM-7: Add Cross-Builder Test Fixtures

Goal: Make future changes safer.

Suggested fixtures:

- One simple weapon.
- One class mod with passive chains.
- One imported known-working modded weapon.
- One imported code with typed tokens.
- One code with array tokens.

For each fixture, store:

- input human serial
- expected Base85 if stable
- expected parser fields
- expected delivery preview behavior
